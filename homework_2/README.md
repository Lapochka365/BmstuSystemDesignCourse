# ТигрБанк — Модуль «Учёт финансов»

## Описание предметной области

Программа моделирует модуль учёта личных финансов банковского приложения ТигрБанк (Московский зоопарк). Пользователь управляет **банковскими счетами** (BankAccount), распределяет операции по **категориям** (Category) и фиксирует **финансовые операции** — доходы и расходы (Operation). Модуль предоставляет аналитику за произвольный период, поддерживает импорт/экспорт данных в форматах CSV, JSON и YAML, а также автоматический пересчёт баланса счёта.

### Доменная модель

- **BankAccount** — банковский счёт с уникальным идентификатором, названием и балансом.
- **Category** — категория операции (доход или расход), например «Зарплата» или «Кафе».
- **Operation** — финансовая операция, привязанная к конкретному счёту и категории, с суммой, датой и необязательным описанием.
- **OperationType** — enum: INCOME (доход) / EXPENSE (расход).

Все сущности поддерживают soft delete через флаг `active`, что сохраняет историю для возможной аналитики и экспорта.

## Архитектура и слои

```
tigrbank/
├── domain/          — доменные классы (BankAccount, Category, Operation, OperationType)
├── repository/      — интерфейсы репозиториев + in-memory реализации
├── service/         — интерфейсы бизнес-сервисов
│   └── impl/        — реализации сервисов
├── io/              — интерфейсы стратегий импорта/экспорта
│   └── strategy/    — реализации (CSV, JSON, YAML)
└── console/         — консольный UI
```

## Реализованная функциональность

### Основная функциональность
- Создание, редактирование (переименование) и удаление (soft-delete) счетов, категорий и операций (доходов/расходов).
- Автоматическое обновление баланса счёта при создании и удалении операции.
- Валидация при создании операции: проверка существования счёта и категории, совпадение типа операции с типом категории, положительность суммы.

### Аналитика
- Подсчёт разницы доходов и расходов за выбранный период.
- Группировка расходов по категориям за период.
- Группировка доходов по категориям за период.
- Неактивные (удалённые) операции исключаются из аналитики.

### Импорт и экспорт данных
- Экспорт всех данных (включая неактивные) в форматы: CSV, JSON, YAML.
- Импорт данных из файлов CSV, JSON, YAML с дедупликацией по ID и автоматическим пересчётом балансов затронутых счетов.

### Управление данными
- Пересчёт баланса счёта на основании всех активных операций — для исправления несоответствий.

### Статистика
- Измерение времени выполнения ключевых операций через аннотацию `@Timed` и `TimingAspect`.

## Принципы SOLID

### Single Responsibility Principle (SRP)

Каждый класс отвечает ровно за одну задачу:

| Класс | Зона ответственности |
|---|---|
| `BankAccountServiceImpl` | Управление счетами (создание, переименование, удаление, пересчёт баланса) |
| `OperationServiceImpl` | Управление операциями и обновление баланса при создании/удалении |
| `CategoryServiceImpl` | Управление категориями |
| `AnalyticsServiceImpl` | Аналитика (разница доходов/расходов, группировка по категориям) |
| `DataExportServiceImpl` / `DataImportServiceImpl` | Координация экспорта/импорта — делегирование логики стратегиям |
| `AccountMenuHandler`, `CategoryMenuHandler` и др. | Каждый обработчик обслуживает ровно один раздел консольного меню |

### Open/Closed Principle (OCP)

Система открыта для расширения, но закрыта для модификации:

- **Стратегии импорта/экспорта:** интерфейсы `DataExportStrategy` и `DataImportStrategy` позволяют добавить новый формат (например, XML), просто создав новый `@Component`-класс. Ни `DataExportServiceImpl`, ни `DataImportServiceImpl` изменять не нужно — новая стратегия автоматически подхватывается через `List<DataExportStrategy>`, инжектируемый Spring'ом.
- **Репозитории:** интерфейс `Repository<T>` позволяет заменить in-memory хранение на БД-реализацию без изменений в сервисном слое.

### Liskov Substitution Principle (LSP)

- `InMemoryBankAccountRepository`, `InMemoryCategoryRepository`, `InMemoryOperationRepository` — взаимозаменяемы с интерфейсами `BankAccountRepository`, `CategoryRepository`, `OperationRepository` соответственно. Любая альтернативная реализация подставляется без изменений вызывающего кода.
- `JsonStrategy`, `YamlStrategy`, `CsvStrategy` — все корректно реализуют контракты `DataExportStrategy` и `DataImportStrategy`.

### Interface Segregation Principle (ISP)

- Сервисы разделены на специализированные интерфейсы: `BankAccountService`, `CategoryService`, `OperationService`, `AnalyticsService`, `DataExportService`, `DataImportService`. Клиентский код (консольные обработчики) зависит только от тех интерфейсов, которые ему действительно нужны.
- Репозитории: общий `Repository<T>` содержит только базовые CRUD-методы; специфичные запросы (например, `findByDateRange`, `findByBankAccountId`) вынесены в `OperationRepository`, а не раздувают общий интерфейс.

### Dependency Inversion Principle (DIP)

- Все сервисы зависят от **интерфейсов** репозиториев (`BankAccountRepository`, `CategoryRepository`, `OperationRepository`), а не от конкретных in-memory реализаций.
- `DataExportServiceImpl` и `DataImportServiceImpl` зависят от интерфейсов `DataExportStrategy` / `DataImportStrategy`, а не от `JsonStrategy`, `CsvStrategy` и т.д.
- Консольные обработчики зависят от интерфейсов сервисов (`BankAccountService`, `OperationService` и др.), а не от `Impl`-классов.

## DI-контейнер (Spring Boot)

Взаимодействие модулей реализовано через DI-контейнер **Spring Boot**. Все зависимости инжектируются через **конструкторы**, что обеспечивает:

- **Явность зависимостей** — все зависимости класса видны в конструкторе.
- **Неизменяемость** — поля зависимостей объявлены как `final`.
- **Тестируемость** — в unit-тестах зависимости подставляются вручную (без Spring-контекста).

Конфигурация:
- Доменные классы — Java объекты без Spring-аннотаций.
- Репозитории — `@Component`.
- Сервисы — `@Service`.
- Стратегии импорта/экспорта — `@Component`.
- Консольный UI — `@Component`, точка входа через `CommandLineRunner`.

## Потенциальные проблемы при расширении

1. **Добавление нового типа сущности** (например, «Бюджет» или «Цель»): потребуется создать новый доменный класс, репозиторий, сервис и консольный обработчик. Также необходимо будет обновить все стратегии импорта/экспорта (CSV, JSON, YAML) и контейнер `FinanceData` для поддержки новой сущности — это нарушает OCP в части import/export.

2. **Переход на реальную БД**: in-memory репозитории не поддерживают транзакции. Операции, включающие несколько шагов (создание операции + обновление баланса), не атомарны. При переходе потребуется добавить транзакционное управление.

3. **Многопользовательский режим**: текущая доменная модель не содержит идентификатора пользователя. Добавление поля `userId` затронет все сущности, репозитории и сервисы.

4. **Масштабирование консольного UI**: при добавлении новых разделов меню класс `ConsoleApp` нужно будет модифицировать (новый case в switch). Переход на паттерн Command упростил бы расширение.

5. **CSV-стратегия** реализована отдельно от Jackson-стратегий и содержит собственный парсер. При усложнении формата (вложенные объекты) её поддержка станет трудоёмкой.

## Почему введённые абстракции улучшают качество дизайна

1. **Интерфейсы сервисов** (`BankAccountService`, `OperationService` и др.) отделяют бизнес-логику от её реализации. Консольный UI не знает, как именно хранятся данные — он работает через абстракции. Это позволяет менять реализацию (например, добавить кэширование) без изменений в UI.

2. **Интерфейс `Repository<T>`** унифицирует CRUD-операции для всех сущностей. Благодаря этому сервисы не зависят от конкретного способа хранения (в памяти, в БД, в файле). Замена `InMemory*Repository` на иную потребует лишь новой реализации интерфейса.

3. **Стратегии `DataExportStrategy` / `DataImportStrategy`** изолируют логику для форматов. Добавление поддержки нового формата (XML, Protobuf) — это создание одного нового класса без изменений в сервисном слое или UI. Spring автоматически подхватит новую стратегию.

4. **Абстрактный класс `AbstractJacksonStrategy`** устраняет дублирование кода между JSON и YAML стратегиями в рамках работы с Jackson. Общая логика сериализации/десериализации написана один раз; конкретные стратегии лишь задают тип `ObjectMapper`.

5. **AOP-аспект `TimingAspect`** выносит сквозную функциональность (измерение времени) из бизнес-логики. Для добавления замера к любому методу достаточно поставить аннотацию `@Timed` — не нужно менять код метода.

6. **Разделение консольных обработчиков** по сущностям (`AccountMenuHandler`, `CategoryMenuHandler` и т.д.) не даёт разрастаться одному файлу с обработкой консоли — каждый обработчик компактен и независим.

## Модульное тестирование

В проекте реализовано **38 unit-тестов** в 6 тестовых классах:

| Класс | Кол-во тестов | Что проверяет |
|---|---|---|
| `DomainValidationTest` | 6 | Валидация доменных объектов (null, blank, отрицательные суммы) |
| `BankAccountServiceTest` | 7 | CRUD счетов, пересчёт баланса |
| `CategoryServiceTest` | 7 | CRUD категорий, soft delete |
| `OperationServiceTest` | 10 | CRUD операций, обновление баланса, валидация типа категории, обработка несуществующих сущностей |
| `AnalyticsServiceTest` | 5 | Аналитика: net income, группировка по категориям, фильтрация по датам, исключение неактивных операций |
| `DataExchangeRoundTripTest` | 3 | Round-trip экспорт → импорт для JSON, YAML, CSV |

Тесты написаны без Spring-контекста — зависимости инжектируются вручную через конструкторы, что подтверждает корректность архитектурного разделения и работу DIP.
